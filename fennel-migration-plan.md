# Fennel/Lua Configuration Migration Plan

## Objective

Replace every exokephalos-owned TOML configuration format and the `yq` custom-action engine with embedded Fennel and Lua.

This is a hard cutover. There are no users to migrate, so do **not** retain TOML compatibility, implement automatic conversion, or add a migration command. Existing workspace configuration must be rewritten as Fennel.

The finished application must:

- Load workspace configuration directly from Fennel.
- Support Fennel and Lua workspace modules.
- Use Fennel predicates for views, subviews, and action applicability.
- Use Fennel/Lua functions for actions instead of yq expressions.
- Keep config execution sandboxed.
- Support per-action, host-local grants for filesystem and network capabilities.
- Preserve CEL only for API-key authorization and the public `POST /api/query/ids` endpoint.
- Remove the Go TOML parser and yq dependencies from the final module graph.

## Existing Architecture and Migration Boundaries

The application is Go-based:

- `internal/config/` currently discovers and parses workspace and local TOML files.
- `internal/filter/` is CEL. It must remain for API keys and `/api/query/ids`, but must no longer power views, subviews, or actions.
- `internal/action/` currently validates and applies yq expressions. Replace it with the Lua action runtime.
- `internal/tui/`, `internal/handlers/`, and the PWA bootstrap all consume `config.Config` and action metadata.
- `internal/syncsvc/` and `internal/tui/sync.go` sync root-level TOML configuration files. They must sync the new workspace Fennel entrypoint and module tree instead.
- The React PWA edits synced workspace configuration through `/api/app/configs`; it does not execute workspace Lua in the browser. Server-side execution is authoritative.

Do not alter the external TOML generated by `xo helix-init` (`.helix/languages.toml`). That is Helix configuration, not an exokephalos configuration dependency.

## Target Filesystem Layout

### Synced workspace configuration

```text
EXO_DIR/
├── exo.fnl                    # required workspace configuration entrypoint
├── modules/
│   ├── views.fnl              # optional Fennel module
│   ├── actions.fnl            # optional Fennel module
│   └── integrations/
│       └── books.lua          # optional Lua module
└── ... markdown notes and assets ...
```

`exo.fnl` is the only workspace entrypoint. It returns a table containing `:default-view`, `:views`, and `:actions`. Modules are loaded with normal Fennel/Lua `require` syntax, but module resolution is limited to `modules/` in the workspace.

Module names map to `modules/<module-name>.fnl` or `modules/<module-name>.lua`; for example, `(require :modules.views)` resolves to `modules/views.fnl`, then `modules/views.lua` if no Fennel file exists. Reject paths containing traversal segments, absolute paths, native modules, and ambient Lua package paths.

### Local-only configuration

```text
EXO_DIR/.exo/
├── tui.fnl                    # TUI/sync settings for this machine
├── serve.fnl                  # server settings for this machine
├── permissions.fnl            # local action capability grants
├── cache.sqlite
├── server.sqlite
└── keys/
```

Never sync files below `.exo/`.

## Fennel Configuration Contract

Use idiomatic Fennel tables and functions. The exact Go structs may remain close to the current types, but the Fennel source contract is authoritative.

```fennel
(local views (require :modules.views))
(local actions (require :modules.actions))

{:default-view :notes
 :views views
 :actions actions}
```

Each view has its existing display fields (`:name`, `:key`, `:show-tags`, `:title-field`, `:subtitle-field`, `:sort-field`, `:sort-order`, `:template`, `:preview-template`, `:stats-template`) and a Fennel predicate:

```fennel
{:notes {:name "Notes"
         :key "n"
         :when (fn [note]
                 (= note.type "note"))
         :template "---\ntype: note\ntags: []\ntitle: {{.Title}}\n---\n"
         :subviews [{:name "All" :when (fn [_] true)}
                    {:name "Todo"
                     :when (fn [note]
                             (and (contains? note.tags "todo")
                                  (not (contains? note.tags "done"))))}]}}
```

The note passed to predicates and actions has at least:

```fennel
{:id "..."
 :path "..."
 :type "note"
 :tags ["tag"]
 :frontmatter {:title "Example"}
 :body "Markdown body"}
```

Predicates must return a boolean. A config source error, runtime error, or non-boolean predicate result must fail config validation/reload with a source-aware error.

## Action Contract and Permission Model

Actions replace yq expressions. Each action has `:description`, optional `:when`, optional `:permissions`, and required `:run`.

```fennel
{:finish-book
 {:description "Mark book as finished"
  :when (fn [note] (contains? note.tags "reading"))
  :run (fn [note]
         (assoc note :frontmatter
                (-> note.frontmatter
                    (update :tags (fn [tags]
                                    (concat (remove #(= $1 "reading") tags)
                                            ["read"]))))))}

 :lookup-metadata
 {:description "Fetch metadata"
  :permissions [:network]
  :run (fn [note exo]
         ;; Use only the capability object passed by the host.
         note)}}
```

`run` receives the complete note and returns its replacement. The Go host validates the result before writing it. It must reject malformed result tables and preserve atomic note writes. Actions may modify frontmatter and body; encrypted-note handling must continue to encrypt only the body after the final replacement note has been validated.

Workspace declarations only request capabilities. They never grant them. The effective capability set is the intersection of:

1. Capabilities declared in the workspace action.
2. Capabilities granted for that action in the local `.exo/permissions.fnl` of the executing host.

Use scoped local grants such as:

```fennel
{:actions
 {:lookup-metadata
  {:network {:origins ["https://openlibrary.org"]}}

  :import-reference
  {:filesystem {:read ["references/**"]
                :write ["generated/**"]}}}}
```

Initial capabilities:

- `:filesystem`: only host-provided helpers, constrained to allowed `EXO_DIR`-relative paths. Reject absolute paths, traversal, and paths outside granted glob roots.
- `:network`: only host-provided HTTPS requests to granted origins, with request and response-size limits plus timeouts.

Do not expose `io`, `os`, `debug`, shell execution, arbitrary file loading, native Lua modules, ambient environment variables, or unrestricted HTTP to configuration/action code.

TUI actions execute using the TUI machine's `.exo/permissions.fnl`. PWA/API actions execute on the server using the server's `.exo/permissions.fnl`. The browser must not run workspace Lua.

## Implementation Checklist

### 1. Establish dependencies and the embedded runtime

- [x] Download and vendor the supplied Fennel 1.6.1 standalone compiler from `https://fennel-lang.org/downloads/fennel-1.6.1` under `internal/config/fennel/`.
- [x] Preserve the compiler's SPDX/license header and add a short provenance/version comment beside the Go embed declaration.
- [x] Add `github.com/yuin/gopher-lua` as a direct Go dependency. It is a Lua 5.1 VM suitable for an in-process Go host.
- [x] Implement a dedicated runtime package under `internal/config/` rather than mixing Lua stack manipulation into the TOML loader.
- [ ] Configure the runtime with only base/table/string/math libraries needed by compiled Fennel. Do not open io, os, debug, package filesystem searchers, or channel libraries.
- [x] Load the compiler once per runtime state and compile Fennel source in-process.
- [x] Add context/time limits to predicate/action evaluation and ensure runtime errors include the workspace module path and action/view name.
- [x] Add unit tests proving the compiler can compile a trivial Fennel table and function before changing config discovery.

### 2. Implement safe module resolution

## Completion Criteria

The migration is complete only when all of the following are true:

- [ ] A fresh workspace configured only with `exo.fnl` works in the TUI, server, and PWA.
- [ ] Workspace config can import sandboxed Fennel and Lua modules from `modules/`.
- [ ] Views, subviews, and actions are evaluated through Fennel/Lua rather than CEL/yq.
- [ ] API-key filters and `/api/query/ids` still use CEL and have unchanged behavior.
- [ ] Action capabilities are deny-by-default and require matching local grants.
- [ ] Synced configuration includes the entrypoint/module graph but excludes `.exo/` local configuration and permissions.
- [ ] No TOML workspace/local configuration loader, TOML Go parser dependency, or yq dependency remains.
- [ ] The full Go test suite and PWA production build pass.
